## 1. 解決すべき現場の課題（Why）

### 現在の介護・福祉施設の実態

介護・福祉施設の現場では、スタッフは本来の支援業務に加えて、
膨大な「紙ベースの記録業務」を毎日こなしている。

具体的には以下の3領域にわたる記録が存在する。

**スタッフの労務管理**
- 出退勤の手書き記録
- シフト表の手作成と変更管理
- 有給・欠勤・遅刻早退の管理
- 月次集計と給与計算のための転記作業

**利用者・入居者の情報管理**
- 入居・利用開始時の初期情報の紙記録
- 個別支援計画書・ケアプランの作成と更新
- 状態変化の記録と引き継ぎ
- 家族への連絡記録

**日々の記録類**
- ケース記録（支援員の日々の観察記録）
- 日報・業務日誌
- 服薬記録・バイタル記録
- インシデント報告書
- 実地指導対応のための各種記録

### 現場が抱える本質的な問題

これらの記録業務は、施設によって異なるソフトウェアで管理されているが、
多くの施設では以下の問題が根深く残っている。

一つ目は二重入力の問題である。紙に書いてからPCに入力するという非効率な作業が常態化している。

二つ目は転記ミスの問題である。手書き→PC入力の過程でヒューマンエラーが発生し、
介護報酬請求ミスや法令違反につながるリスクがある。

三つ目はリアルタイム性の欠如である。紙記録は後からまとめて入力するため、
情報の鮮度が失われ、チーム間の引き継ぎに支障をきたす。

四つ目はITスキル格差の問題である。現場スタッフの年齢層が高く、
複雑なPCソフトウェアの操作を苦手とするスタッフが多い。

五つ目は法的要件への対応負荷である。実地指導で求められる記録の質・量・
フォーマットを満たすために、多大な残業が発生している。

### North Star OSが実現する世界

「現場スタッフが記録業務から解放され、本来の支援・介護に集中できる施設」

スタッフはスマホに向かって話しかけるか、紙を写真に撮るか、
タブレットをタップするだけで、AIが法的に求められる完全な記録を自動生成する。

## 2. 解決策の基本概念（What）

### マルチ入力方式によるデータ化

現場の多様な状況に対応するため、以下の3つの入力方式を並列で提供する。
どの方式でも同じ品質のアウトプットが得られることが絶対条件。

```
【入力方式1】スマホ音声入力
     ↓
【入力方式2】紙・PDF OCR（写真撮影）
     ↓
【入力方式3】PC・タブレット直接入力
     ↓
     ↓ ← 3方式すべてが同じパイプラインに流れ込む
     ↓
【Supabase統合DB（東京リージョン）】
     ↓
【AIによる自動処理・帳票生成】
     ↓
【必要な出力をボタン一つで生成】
```

---

## 3. 3つの入力方式の詳細設計

### 入力方式1: スマホ音声入力（現場の「動」の記録）

**対象シーン**
- 支援中・介護中にその場でメモできない瞬間の記録
- 利用者の様子を観察しながらリアルタイムで記録したい場面
- 移動中・手が塞がっている状態での記録

**仕組み**
スタッフがスマホのマイクボタンをタップして話しかける。
「えーっと今日のAさんについてです。午前中はシール貼りの作業で...」
という口語・雑音混じりの音声を、AIが以下の処理を行う。

1. Web Speech APIまたはWhisper APIで高精度テキスト化
2. フィラー（えーっと・あのー等）を自動除去
3. SOAP形式（S/O/A/P）の正式ケース記録フォーマットに整形
4. 実地指導基準を満たす公用文に変換
5. Supabaseに保存

**UI基準（絶対厳守）**
- ボタンは画面の80%を占める大型マイクボタン1つのみ
- 文字サイズ18px以上
- タップ領域44px以上
- 高コントラスト（4.5:1以上）
- 1画面1アクション（迷わせない）
- 完全スマホファースト・片手操作

### 入力方式2: 紙媒体OCR・PDF（既存の「紙習慣」をそのままデジタル化）

**対象シーン**
- 手書きの出勤簿・タイムカードのデジタル化
- 既存の紙の記録用紙をそのまま使いたい施設
- FAX・PDFで届く書類の処理
- 利用者の写真付き記録

**仕組み**
スタッフがスマホのカメラで紙を撮影するだけ。
AIが以下の処理を自動実行する。

1. HEIC/JPG/PDF形式を受け取る
2. OCRエンジン3段フォールバックで文字認識
   - 第一優先: Mac Vision Framework（無料・最速）
   - 第二優先: Claude API（手書き解読最強）
   - 第三優先: Gemini API（表構造解析に強い）
3. 認識結果を構造化データに変換
4. 元の紙フォーマットと照合してエラーチェック
5. Supabaseに保存

**この方式の価値**
施設が今使っている紙の記録用紙を変える必要がない。
現場の「紙習慣」を壊さずにデジタル化できる。
スタッフへの教育コストがほぼゼロ。

### 入力方式3: PC・タブレット直接入力（事務室での「静」の作業）

**対象シーン**
- 管理者・サービス管理責任者（サビ管）による計画書作成
- 月次の集計・分析作業
- 複雑な個別支援計画・ケアプランの策定
- 請求データの確認・修正

**仕組み**
ウェブブラウザで動作するシンプルなフォーム。
入力補助として以下を提供する。

1. 過去の記録から自動サジェスト
2. 法的要件チェックのリアルタイム警告
3. フォーマット自動整形
4. Supabaseへの即時同期

## 4. Supabase統合DBパイプライン

### データの流れ

3つの入力方式から入ったデータはすべてSupabase（東京リージョン）に集約される。

```
音声入力 ──────────────────────────┐
紙・PDF OCR ────────────────────── ├──→ Supabase（統合DB）──→ AI処理 ──→ 出力
PC直接入力 ─────────────────────── ┘
```

### 4つの出力領域

**[実務]出力**
- ケース記録（SOAP形式・実地指導対応）
- 日報・業務日誌
- 申し送り・引き継ぎ書
- インシデント報告書

**[法規制]出力**
- 介護配置基準（3:1等）の自動照合・アラート
- A型事業所の労働時間スコア算出
- 人員配置基準の充足確認
- 処遇改善加算の算定根拠書類

**[経営]出力**
- A型収支予測レポート
- 収益改善のAI提言
- 稼働率・利用状況分析
- スタッフ労働時間の最適化提案

**[請求]出力**
- 国保連請求データのドラフト作成
- 介護給付費明細書の自動生成
- 加算算定根拠の自動チェック

## 5. 対象施設・サービス種別

### 主要ターゲット

**特定施設入居者生活介護**
- 入居者の日々の記録（ケース記録・バイタル・服薬）
- シフト管理と人員配置基準（3:1）の自動照合
- 処遇改善加算の根拠書類

**就労継続支援A型事業所**
- 利用者の個別支援計画と日々の作業記録
- 労働時間スコアの自動算出（A型特有の要件）
- 工賃・賃金計算の補助

**その他介護・障害福祉サービス**
- 各サービス種別に応じたフォーマット対応
- 実地指導に耐えうる記録品質の担保

## 7. 開発ロードマップ

### Phase 1: プロンプト開発（費用ほぼ0円）

ChatGPTカスタムGPTsまたはclaude.aiで以下を検証する。

- ケース記録自動生成プロンプトの精度検証
- SOAP形式変換の品質確認
- 各施設種別のフォーマット対応

既存プロンプト: Prompt_CaseRecord_TypeA.md（GitHubに保存済み）

### Phase 2: MVPアプリ化

- フロントエンド: Vite（React）/ Tailwind CSS
- ホスティング: Vercel（無料枠）
- DB・認証: Supabase（無料枠）
- 音声認識: Web Speech API（無料）→ 高精度化時にWhisper API
- AI処理: Claude API / GPT-4o mini
- デプロイ: Vercel（GitHub Push → 自動デプロイ）

### Phase 3: 顧客テスト

既存コンサル先・代行先の少数施設に無料または低額で使ってもらい、フィードバックを取得する。

## 9. 技術的注意事項

**個人情報保護**
- 利用者名・スタッフ名等の個人情報はSupabase内で暗号化
- 外部AI APIに送信する際は識別子を匿名化
- データは日本国内（東京リージョン）のみに保存

**オフライン対応**
- 施設内のWi-Fi環境が不安定な場合を想定
- PWA（Progressive Web App）によるオフラインキャッシュ対応
- 通信回復後に自動同期

**既存ソフトとの連携**
- ワイズマン・カイポケ等の既存介護ソフトへのコピペ出力を優先
- CSV出力による既存システムへのデータ移行サポート
- 段階的な移行を支援（いきなり全置換しない）
