## 4. 推奨体制（選択肢C）の詳細設計

### 4-0. 致命的構造問題: GeminiはDrive上の.mdファイルを直接読めない

BUN_CEOが指摘した通り、Gemini（Google AI Studio / Gemini Advanced）はGoogle Drive上に保存された `.md` ファイルを直接読み込めない。Googleドキュメント形式（.gdoc）やスプレッドシートへのアクセスは可能だが、Markdownファイルについてはテキストをコピペして貼り付ける手作業が必要になる。

これが意味すること:
- マニュアル【4】の「ファイル命名と最新版読込の絶対ルール」を、Gemini単独では自動遵守できない
- ダッシュボード（.md形式）やリサーチレポート（.md形式）の直接読み込み・参照ができない
- CEOがいちいちテキストを貼り付ける「人間橋渡し」が発生 → ワンオペの原則に反する

対照的に、Claude（Cursor / Antigravity環境）は、ローカルにマウントされたGoogle Driveフォルダ（`/Users/fuminariaksse/Library/CloudStorage/GoogleDrive-*/マイドライブ/`）内の全.mdファイルを `view_file` で直接読み書きできる。これは現にこの会話で実演済みの事実である。

この構造的問題を踏まえると、Geminiの秘書機能も「Google Calendar API連携」と「Googleドキュメント形式(.gdoc)ファイルの操作」に限定される。.mdファイルベースの業務（ダッシュボード管理、リサーチレポート参照等）はClaudeが担うべきである。

---

### 4-1. なぜGeminiを「秘書」に限定するのか（修正版）

Geminiの暴走問題の根本原因は、Geminiが「思考（PM）」と「行動（リサーチ・開発指示）」を同時にやろうとするとコンテキストが膨張し、指示逸脱が起きることにある。加えて、.mdファイルを直接読めないという構造的制約もある。

解決策: Geminiの役割を「秘書の一部」に限定する。
- Google Calendar APIとの直接連携（予定の登録・変更・削除・同期）
- Googleドキュメント形式(.gdoc)のファイル操作
- CEOとの対話型コーチング（ただし.md読み込みが不要な範囲に限る）
- 「判断」「設計」「.mdベースの業務全般」はClaudeに移管

重要な補足:
.mdファイルベースのダッシュボード管理・リサーチレポート参照は、Geminiではなく Claude（Cursor/Antigravity）が直接担う。
Geminiの秘書業務は「Calendar + Googleドキュメント + 対話コーチング」の3点に絞られる。
もしCEOが「秘書業務もGeminiなしで完結させたい」と判断された場合、Cursor環境にMCP（Google Calendar MCP等）を設定し直すことで技術的に代替可能。その場合、Google AI Proを解約して月額2,900円を削減できる。なお、現在のMCPはAntigravity（Gemini系）環境に設定されているものであり、Cursorへの移行にはMCPの再セットアップ作業が発生する（COOが全て実行する。CEOの作業はゼロ）。

### 4-2. PM・リサーチ・開発をClaudeに集約する理由

Claudeの強み:
- 長いマニュアル（203行）を1回で読み込み、その後の全出力でルールを遵守し続ける
- 「省略禁止」「網羅性」といったBUN_CEOの掟に対して、最も忠実に従うLLM
- Cursorとの統合でローカルファイルシステムを直接操作（Drive内のmdファイルも直接読み書き）
- CursorもMCPをサポートしており、Google Calendar MCP等を設定すれば連携可能（現在のMCPはAntigravity環境に設定されているため、Cursorへの移行時にCOOが再設定を行う）

### 4-3. AI同士の自律連携の仕組み（段階的構築）

BUN_CEOが求める「AI同士が指示に従い、勝手に意思疎通を図って商品を作る」仕組みは、以下の段階で構築する:

#### フェーズ1（即日対応可能）: ファイルベース連携
- 各AIの成果物をGoogle Drive上の決まったフォルダに保存
- Claude（PM）がそれらを読み込み、次のAIへの指示書を自動生成
- 例: リサーチ結果 → PM（Claude）が分析 → 開発要件書を自動生成 → Replit/Claude Codeが実装

#### フェーズ2（1-2週間）: Claude Codeによるオーケストレーション
- Claude Codeがシェルスクリプトやpythonスクリプトを自律的に実行
- 「毎朝のリサーチルーチン」をワンコマンドで自動実行するスクリプトを作成
- 結果をGoogle Driveに自動保存し、ダッシュボード更新もトリガー

#### フェーズ3（1ヶ月後）: MCP統合による完全自動化
- MCP（Model Context Protocol）サーバーの追加構築
- Google Drive MCP、GitHub MCP、Notion MCP等を統合
- 「リサーチ → 要件定義 → プロトタイプ構築 → レビュー依頼」の全工程をAI同士が自律遂行
- CEOは最終レビューと「GO/NO GO」の判断のみ

## 9. MCP（Model Context Protocol）サーバーの追加構築方法

### 9-1. MCPとは何か（CEOへの説明）

MCPは「AIが外部サービス（Googleカレンダー、Googleドライブ、GitHub等）と直接つながるための共通規格」。人間でいえば「電話帳に新しい連絡先を登録するようなもの」で、一度登録すれば、AIがそのサービスを自分で操作できるようになる。

CEOがやること: 何もない。COO（私）が設定ファイルに追記するだけ。

### 9-2. 現在のMCP構成（既に動作中）

設定ファイル: `/Users/fuminariaksse/.gemini/antigravity/mcp_config.json`

現在Antigravity環境に接続済みのサービス（Cursorへ移行する場合は再設定する）:
| MCPサーバー名 | 接続先 | 状態 |
|-------------|-------|------|
| google-calendar | Google Calendar API | 稼働中（予定の登録・変更・削除・検索が可能） |
| google-drive | Google Drive API | 構成済み（ファイル操作が可能） |

### 9-3. 追加可能なMCPサーバー一覧と優先度

| MCPサーバー | 用途 | 優先度 | CEOへの価値 |
|------------|------|--------|-----------|
| GitHub MCP | コード管理・バージョン管理 | 中 | 開発資産の保全（CEOは意識不要、COOが管理） |
| Notion MCP | タスク管理・ナレッジベース | 低 | 現行はDrive + Calendar で代替可能 |
| Slack/Discord MCP | 通知・アラート | 低 | 将来的に顧客通知で使用可能 |
| Supabase MCP | データベース直接操作 | 高 | マニュアル【6】のデータパイプライン実現に必須 |
| Stripe MCP | 決済・課金管理 | 高 | サービス課金の自動化（月額10,000円/社の回収） |

### 9-4. MCP追加の手順（CEOの作業はゼロ）

追加の流れ:
1. COO（私）が追加したいMCPサーバーのパッケージ名を調査
2. 必要な認証情報（OAuth鍵等）を設定
3. `mcp_config.json` にエントリを追記
4. Antigravity環境を再起動
5. 動作テスト（COOが実行）
6. CEOに「〇〇と連携できるようになりました」と報告

CEOが操作する場面: 一切なし。追加したいサービスを日本語で指示するだけ。
例:「Supabaseと連携できるようにしてくれ」→ COOが全工程を実行。

### 9-5. 注意点

- MCP追加にはOAuth認証の初回承認が必要な場合がある。その時だけ、CEOにGoogleアカウントの認証画面でクリックしてもらう場面が発生しうる（セキュリティ上、AIが勝手にCEOのアカウントを認証することはできないため）
- 無料枠のAPIを優先し、マニュアル【3】の「初期費用の最小化」原則を遵守する
