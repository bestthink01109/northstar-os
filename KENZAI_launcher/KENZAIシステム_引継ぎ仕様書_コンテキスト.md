## 1. プロジェクト概要

* **システム名**: KENZAI（給与計算・出勤簿全自動化統合システム）
* **目的**: 3つのグループ会社（平野工業、福岡プラント機工、純青）のバラバラな勤怠入力形式（Excel、手書きPDF、手書き写真）をAI/プログラムで一括吸収し、最終的に「元のExcel形式の出勤簿（手書きの補完）」と「税理士へ渡す共通CSV（給与らくだ用）」を同時生成する。
* **BUN_CEOの経営理念・ルール**:
  * **「ALIGN FIRST」「手抜きをしない、網羅する。魂を込める。」**
  * **現場至上主義:** 現場で働くITが苦手なスタッフや、BUN_CEO自身が「コードを一切触らず、直感的にクリック一つで動かせる」こと。
  * 職務放棄（コードの省略、「ここは実装してください」等の手抜き）は絶対に許されない。

## 3. 実装の進捗と会社の仕様

### ① 平野工業（進捗度: 100% / 実データテスト待ち）

* **入力形式**: 各社員別で保管される「出勤簿・有給管理表（Excel）」
* **労務計算ロジック（開発計画書完全準拠）**:
  * **基本の労働時間（勤務体系の前提）**: 本システムの全計算は、以下の勤務体系を前提に構築されています。
    * **一般社員（平野珠美氏以外の全員）**
      * 月曜〜金曜：所定労働時間 7時間（8:00〜17:00、休憩2時間）
      * 土曜日：通常出勤扱い 5時間（8:00〜15:00、休憩2時間）
      * 週所定労働時間：7時間 × 5日 ＋ 5時間 × 1日 ＝ 40時間（ピッタリ）
      * 日の法定労働時間上限：8時間
    * **平野珠美氏（特例）**
      * 月曜〜金曜：所定労働時間 8時間
      * 土曜日：通常出勤扱い（他社員と同様）
      * 週所定労働時間：8時間 × 5日 ＝ 40時間
  * **休憩の動的控除**: 10:00〜10:30、12:00〜13:00、14:00〜14:30の3区間で、実際の就業時間に応じた厳密な控除を自動計算する。
  * **残業の2階層分類と相殺（週40時間）ロジック**:
    * 【法定外残業】1日8時間を超える部分。発生した日に即座に確定し、他日の欠勤や有給との相殺は一切行わない。
    * 【法定内残業（延長）】1日の所定時間〜法定時間（8時間）の間の部分。
    * 【WeeklyAllocatorの根幹技術】各日の「欠勤・有給・遅刻早退による不就労時間（ショート分）」のみを純粋に足し合わせ、週40時間上限への「バッファ枠」を形成する。法定内残業（延長）はこのバッファ枠に収まる分だけが適用され、枠をオーバーした時間は自動的に「法定外残業」へ昇格する設計。
* **出力形式**:
  * アナログのフォーマットを100%維持した **Excelクローン出力**
  * システムの計算と現場の入力に一箇所でもズレがあればあぶり出す **差異レポート.xlsx**

### ② 福岡プラント機工（進捗度: 実データフォーマット検証待ち）

* **入力形式**: 手書きの日報（PDF）
* **処理仕様**: OCRで生テキスト/構造化データを抜き出し、共通フォーマットへパース。

### ③ 純青（進捗度: 実データ検証待ち）

* **入力形式**: 出勤簿の画像（JPG/PNG等）
* **処理仕様**: 全社フォーマットの表写真から出退勤、有給、外勤フラグ（○印）を抽出。

## 4. コア機能のハイライト

1. **社員マスターのCSV動的化**:
   各社のフォルダ内にある `employees.csv` から情報を毎回動的に読み込む。これにより、退職や新入社員の追加、名前の変更は、BUN_CEOがExcelでCSVを書き換えるだけでシステムに即時反映される。
2. **最強のマルチバックエンドOCRエンジン (`ocr_engine.py`)**:
   手書き画像を読み取るため、以下3つのOCRエンジンを搭載し、自動でフォールバック（失敗したら次を試す）する仕組みを実装済み。
   * 第一優先: Mac純正 Vision Framework（無料・ローカル・最速）
   * 第二優先: Claude API (Anthropic - MLLMで手書き解読最強)
   * 第三優先: Gemini API (Google - 表構造解析に強い)

## 5. 次期AIへの引き継ぎ（残タスク）

次にこのプロジェクトを受け取るAIは、以下のタスクから着手すること。

1. **実データテスト（平野工業）**
   * BUN_CEOが指定のGoogleドライブに上げた平野工業の「実際の入力Excelファイル（202604等）」を読み込ませ、計算結果と差異レポートを出力し、BUN_CEOと答え合わせを行う。
2. **OCR精度のチューニング（福岡機工・純青）**
   * 実データ（PDF・書き込み写真）を受領後、搭載済みのOCRエンジンを切り替えながらテスト実行。
   * もし文字化けや座標ズレがあれば、`ocr_parser.py` の抽出用プロンプト（フォーマット指定）や後処理の正規表現をチューニングすること。
3. **APIキーのセキュアな設定**
   * ClaudeやGeminiのAPIが動作するための環境変数（`ANTHROPIC_API_KEY` 等）を BUN_CEOの環境に設定するサポートを行う。
