## 各ツールの役割分担

| ツール | 役割 | やること | やらないこと |
|-------|------|---------|------------|
| **NS-OS** | 実行・生成 | タスク実行・成果物生成・自動化 | 長期知識蓄積・文書横断検索 |
| **Obsidian** | 蓄積・連結 | 知識の永続化・ノート連結・日次管理 | 大量文書の一括読み込み |
| **NotebookLM** | 分析・検索 | 大量文書の横断分析・引用付き回答 | 継続的なノート管理・タスク実行 |

## 情報フロー（5つのパターン）

### フロー1：法令研究
```
RSCリサーチ（n8n毎日6:00）→ references/に追記
  → Obsidian 02_Resources/介護法令/にコピー
  → NotebookLMに介護法令PDFと一緒にアップロード
  → 「2024年処遇改善加算改正で特定施設の加算率はどう変わったか？」
  → 引用付き回答 → Obsidian 04_Inbox/にメモ保存
  → arscontextaが関連法令ノートと自動連結
```

### フロー2：月次P&L分析
```
FINレポートWF（n8n毎月1日）→ FIN月次レポート生成
  → Obsidian 03_Archives/にコピー
  → NotebookLMに直近12ヶ月分をアップロード
  → 「過去6ヶ月の労働分配率の傾向と今月の異常値の原因仮説は？」
  → 引用付き回答 → P&L 4ステップ Step2に使う
```

### フロー3：COO_Contextセカンドブレイン化
```
毎日のセッション終了 → COO_Context生成
  → Obsidian 03_Archives/COO_Context_履歴/にコピー
  → arscontextaが自動でノート連結
  → 「3ヶ月前にKENZAIの開発方針を決定した時の根拠は？」→即座に引き出せる
  → 累積50件以上になったらNotebookLMにもアップロード
```

### フロー4：競合・市場分析
```
競合資料・業界レポートを収集
  → NotebookLMにアップロード
  → 「介護事業所の人材確保における競合の差別化戦略を列挙して」
  → 引用付き分析 → Obsidian 01_Areas/BizDev/にノート保存
  → BizDevスキルのICEスコア評価に使う
```

### フロー5：朝のセッション準備
```
朝7:00 → Obsidianデイリーノートを開く
  → Dashboardを確認してタスク転記
  → Claudian起動：「今日の最優先タスクを前回COO_Contextから整理して」
  → arscontextaが関連ノートを自動提示
  → NS-OSセッション開始（Claude Code本体）
```

## NotebookLM Source設計

### 常設Source
| Source | 更新タイミング |
|--------|-------------|
| 介護法令PDF群（厚労省・福岡県・熊本県） | 法改正時 |
| 処遇改善加算通知 最新版 | 年1回（改正時） |
| 障害福祉サービス等報酬告示 | 改定時（3年ごと） |

### 期間限定Source（目的別に入れ替え）
| Source | 使用目的 | 運用 |
|--------|---------|------|
| FIN月次レポート 直近12ヶ月 | P&L趨勢分析 | 毎月追加・古いものは削除 |
| COO_Context 直近6ヶ月 | 意思決定の棚卸し | ローリング更新 |
| 競合調査レポート群 | 市場機会評価 | BizDev分析時に都度入れ替え |
| 集団指導資料（福岡・熊本） | 法令解釈確認 | 年1回・最新版に差し替え |
